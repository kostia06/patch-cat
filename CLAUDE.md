# Patch — Project Context

This file is read by Claude Code at the start of every session. It is the source of truth for the project's architecture, stack, and conventions. If a task prompt conflicts with this file, this file wins.

## What we're building

Patch is an **MCP (Model Context Protocol) server** that gives any AI assistant — Claude Desktop, Cursor, Claude Code, Windsurf — a permanent, growing toolbox of executable Python tools. When the host AI lacks a capability, Patch generates a Python tool on demand, tests it in an e2b sandbox, persists it to the user's local toolbox, and registers it dynamically with the host so the AI can call it natively from then on.

Eventually, users can opt in to a community registry where tools generated by other Patch users are pulled instead of regenerated. The registry self-evolves: it merges duplicate tools and composes primitives into higher-order tools as it observes usage patterns. Every tool execution produces a forensic audit trail.

## What Patch is not

- **Patch is not the agent.** The host AI (whatever's running in Claude Desktop / Cursor / etc.) is the agent. It has its own LLM, its own context window, its own planner. Patch only exposes capabilities to that host. We do not write agent loops, planner prompts, or context-window management.
- **Patch is not a UI product.** There is no web dashboard, no chat interface, no settings panel for v0.x. The product surface is the MCP server itself plus a docs site (later).
- **Patch is not a knowledge base.** It stores executable tools, not text answers. We do not store explanations, Q&A, or documentation as registry entries.

## Distribution

Single command, no install: `npx -y @patch-cat/mcp`. Users add this to their host AI's MCP config and that's it. No accounts, no API keys for v0.1.

## Stack — locked, do not deviate without flagging

### Local MCP server (the npx package)

- **Language:** TypeScript, Node 20+
- **MCP framework:** `@modelcontextprotocol/sdk` (Anthropic's official TS SDK)
- **Transport:** stdio (the way Claude Desktop / Cursor launch local MCP servers)
- **Tool generator LLM:** `@anthropic-ai/sdk`, model `claude-opus-4-7`
- **Sandbox:** `@e2b/code-interpreter` (TS SDK)
- **Manifest parsing:** `js-yaml`
- **Filesystem paths:** `env-paths`
- **Logging:** `pino`, always to stderr (stdout is reserved for MCP protocol)
- **Build:** `tsup` (single ESM bundle for the bin)
- **Tests:** `vitest`
- **Lint/format:** `biome`

### Generated tool format (Python)

- Python 3.12, executed **only** in e2b sandboxes — never in-process, never via child_process on the host machine.
- Each tool is a single .py file with YAML frontmatter at the top inside `# ---` markers (so the file is still valid Python).
- Frontmatter declares: name, version, description, inputs (with `tainted_ok` flag per input), outputs, capabilities (network/filesystem/human_confirm), runtime (python_version, packages), generated_by, generated_at.

### Registry (Phase 2 — scaffold so it's not blocked, but not built in Phase 1)

- **Compute:** Cloudflare Workers, TypeScript, Hono router
- **Database:** Neon (Postgres) with `pgvector` for semantic search, 768-dim vectors
- **Object storage:** Cloudflare R2, immutable, content-addressed (SHA-256)
- **ORM:** Drizzle
- **Embeddings:** Workers AI `@cf/baai/bge-base-en-v1.5` (768-dim)
- **Quarantine LLM (Phase 3):** Workers AI Llama 3.3 70B
- **LLM gateway:** Cloudflare AI Gateway in front of all LLM calls

### Auth

- **OAuth for tools that call external APIs (Phase 3):** Arcade.dev, runtime tool registration via their `app.tools.add()` API
- **GitHub OAuth for registry contributors (Phase 2):** required to write to the registry; not required to read or use Patch locally

### Observability (Phase 4)

- Langfuse Cloud (free tier) for trace tree
- Cloudflare AI Gateway logs for cost/latency/cache

## Architectural rules — non-negotiable

1. **TypeScript orchestrates, Python executes.** Generated tools are always Python files run in e2b. The TS code never executes generated logic in-process. We do not use `eval`, `Function()`, `vm`, or `child_process` to run untrusted code on the user's machine.

2. **Patch is not the agent.** Do not build agent loops, planner prompts, conversation memory, or context-window management. The host AI does that. Patch only responds to MCP requests.

3. **Capability scopes are enforced by the sandbox, not by prompts.** Tool manifests declare network/filesystem permissions. e2b sandbox configuration enforces them. Never use the LLM as the security boundary.

4. **Untrusted text never reaches the planner.** This becomes critical in Phase 3, but design data flow now so it's possible: any external content (web fetch results, file contents, untrusted args) flows through the quarantine LLM before it reaches a context that can call tools.

5. **All persistent identity is content-addressed.** Tool source blobs in R2 are addressed by SHA-256 of their content. Manifests are immutable; new versions are new content hashes. This makes everything in Phase 4 (forensic audit trails) free to add later.

## Manifest format — locked contract

Every tool is a single Python file. The manifest is YAML in commented frontmatter at the top, between `# ---` markers:

```python
# ---
# name: extract_pdf_table
# version: 1.0.0
# description: Extract tables from a PDF file as JSON.
# inputs:
#   - name: pdf_path
#     type: string
#     description: Absolute path to the PDF file.
#     tainted_ok: true
# outputs:
#   type: array
#   items: object
# capabilities:
#   network: false
#   filesystem: read-only
#   human_confirm: false
# runtime:
#   language: python
#   python_version: "3.12"
#   packages: ["pdfplumber==0.11.4"]
# generated_by: claude-opus-4-7
# generated_at: 2026-05-04T12:34:56Z
# ---

import pdfplumber
import json
import sys

def main(pdf_path: str):
    with pdfplumber.open(pdf_path) as pdf:
        return [page.extract_tables() for page in pdf.pages]

if __name__ == "__main__":
    args = json.loads(sys.stdin.read())
    print(json.dumps(main(**args)))
```

The TS side parses the frontmatter (lines between `# ---` markers, strip leading `# `, parse as YAML, validate against the zod schema in `src/manifest.ts`).

The manifest's `inputs` section maps directly to a JSON Schema object that's exposed as the MCP tool's `inputSchema`.

## Toolbox storage layout

Cross-platform via `env-paths('patch-cat')`:

- macOS: `~/Library/Application Support/patch-cat/`
- Linux: `~/.config/patch-cat/`
- Windows: `%APPDATA%\patch-cat\`

Inside that directory:

- `tools/` — one .py file per tool, manifest + source
- `index.json` — local index: tool name → file path, version, embedding (or null until embedded), last-used timestamp
- `runs/` — per-execution audit logs (Phase 4; create the directory but leave empty in Phase 1)
- `config.json` — user config (API keys, registry endpoint, opt-ins)

## Three meta-tools Patch always exposes over MCP

1. **`patch.generate_tool(description: string)`** — host AI calls this when it wants a capability it doesn't have. Patch generates, sandboxes, persists, registers. Returns the new tool's name. After this call, the new tool is callable directly as its own MCP tool.

2. **`patch.run_tool(name: string, args: object)`** — explicit invocation, mostly for debugging. The auto-exposed individual tools are the normal path for the host to call them.

3. **`patch.list_tools()`** — returns metadata about every tool in the local toolbox (and later, summary of what's available in the registry).

In addition: every tool in the local toolbox is exposed as its own first-class MCP tool, dynamically registered. When a new tool is added (or pulled from the registry), Patch sends `notifications/tools/list_changed` over MCP so the host refreshes its tool list. This is the critical capability — without it, the loop doesn't work.

## Code conventions

- **Strict TypeScript.** `tsconfig` strict mode, no implicit any, no unused locals. ESM only.
- **Modules over classes.** Use classes only when state is genuinely instance-scoped (e.g. an MCP server instance).
- **Errors as values at boundaries.** Use zod for input validation. Inside a module, throw is fine. At module boundaries (especially the MCP request handlers), errors are caught and converted to typed responses.
- **All async, no callbacks.** No `util.promisify`, no callback-style APIs.
- **Logging via pino to stderr.** stdout is reserved for the MCP protocol. Any accidental stdout write breaks the host.
- **No `any`.** If you need to escape the type system, use `unknown` and narrow.
- **No global mutable state.** Pass dependencies through function arguments or a constructed `App` object.

## Things explicitly NOT in v0.1 — resist these

- Web UI / dashboard / settings panel
- Streaming tool output back over MCP (return-then-respond is fine for v0.1)
- Multi-user / accounts
- Registry (Phase 2)
- Versioning, refactoring, or composition of tools
- Browser-based tool execution
- A standalone agent loop or chat interface
- Tool sharing between Patch instances directly (peer-to-peer)
- A CLI for end users (the MCP server is the only surface)

If a task in a kickoff prompt seems to require any of the above, stop and ask before implementing.
